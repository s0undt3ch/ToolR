# Auto-rebuild stale manifest on dispatch

**Date:** 2026-05-21
**Status:** design

## Problem

`toolr <user-command>` and `toolr --help` load
`tools/.toolr-manifest.json` verbatim with no freshness check. When
the user edits, adds, or removes a `tools/*.py` file, the CLI keeps
serving the previous manifest until either tab-completion fires (which
*does* rebuild on hash mismatch) or the user runs
`toolr project manifest rebuild` explicitly. Concretely: adding
`tools/example.py` with a `command_group("example", …)` and four
commands does not make `toolr example --help` work — it still errors
with `unrecognized subcommand 'example'`. This is surprising and a
real papercut for anyone iterating on commands.

The fix is to detect staleness on dispatch and rebuild — cheaply when
only `tools/` changed, slightly more expensively when a third-party
plugin's `toolr-manifest.json` was added/removed/modified — and to
persist the result so subsequent invocations pay nothing.

This work also deprecates and removes the legacy entry-point-based
dynamic discovery (`importlib.metadata.entry_points(group="toolr.commands")`)
that today is the only reason `Manifest.dynamic_hash` exists in its
current "sorted dist-info" form. Third-party plugins must instead
ship a static `toolr-manifest.json` generated by the already-existing
`toolr self build-manifest` command.

## Goals

- Editing, adding, or removing `tools/*.py` is reflected on the very
  next `toolr <user-command>` or `toolr --help` invocation, with no
  manual rebuild step.
- Installing / upgrading / removing a third-party plugin (one that
  ships its own `toolr-manifest.json`) is reflected on the next
  dispatch.
- The dispatch hot path runs entirely in Rust — no Python subprocess
  is spawned just to check freshness or rebuild.
- A failed rebuild does not block dispatch. The user sees a warning
  naming the offending file and falls back to the cached manifest.
- Plugin authors have one documented, mechanical way to ship a
  manifest (`toolr self build-manifest`), and one CI gate to keep
  it fresh (`toolr self build-manifest --check`).

## Non-goals

- Tab-completion freshness behavior is unchanged. The completion
  hot path stays in-memory-only and does not persist.
- The `tools/*.py` dynamic introspection in `_introspect.py`
  (`_import_tools_modules`) is preserved. Users who use dynamic
  patterns the AST parser can't see still benefit from
  `toolr project manifest rebuild`. Dispatch-time freshness does
  not re-introspect Python; runtime-only dynamic patterns require
  the explicit rebuild, same as today.
- No `SCHEMA_VERSION` migration. The toolr CLI rewrite has not
  shipped a release yet, so the on-disk manifest layout can change
  without backward-compat handling.

## Design

### Two-axis freshness signal

A fresh dispatch needs to detect two kinds of change:

1. **Local-tools change** — a `.py` file in `tools/` was added,
   removed, or modified. Already covered by the existing
   `static_hash` (content-based via `hash_tools_dir`).
2. **Third-party-plugin change** — a `toolr-manifest.json` file
   under `<tools-venv>/lib/python*/site-packages/*/` was added,
   removed, or modified.

For (2) the existing `Manifest.dynamic_hash` semantics (sorted-
dist-info hash) are replaced with a **targeted third-party signal**:

```text
third_party_hash = blake3(
  for each path in sorted(glob "<venv>/lib/python*/site-packages/*/toolr-manifest.json"):
    path \0 content_hash(path) \0
)
```

Replacing dist-info hashing means `uv pip install some-unrelated-pkg`
no longer invalidates the manifest. Only changes to packages that
actually contribute toolr commands trigger a rebuild.

The `Manifest` model renames the field: `dynamic_hash` → `third_party_hash`.

### Freshness verdict and rebuild matrix

A new module `crates/toolr-core/src/freshness/` exposes:

```rust
pub enum FreshnessVerdict {
    Fresh,
    StaticDrift,
    ThirdPartyDrift,
}

pub fn compare(cached: Option<&Manifest>, tools_dir: &Path, venv_dir: Option<&Path>)
    -> Result<FreshnessVerdict>;
```

The verdict variants are ordered by "stronger rebuild needed."
When both axes drift simultaneously, `compare` returns
`ThirdPartyDrift` because the third-party rebuild path
(`build_static_manifest_with_venv`) is a superset that also
re-parses `tools/*.py`. There is no `Both` variant — collapsing
into the stronger verdict keeps the call sites' match arms small.

Both the new dispatch-time freshness step and the existing
`complete::resolve_manifest_at_tab` call `compare` and then branch
on what to do with the result. "Preserve dynamic entries" below
means: keep every entry whose `origin` field equals `dynamic`
(produced by the Python introspection pass and not re-runnable
without spawning Python). Static and third-party entries are
re-derived from the fresh rebuild.

| Verdict             | Rebuild action                                                                                                  | Persist? |
|---------------------|-----------------------------------------------------------------------------------------------------------------|----------|
| `Fresh`             | none — use cached as-is                                                                                          | no       |
| `StaticDrift`       | `build_static_manifest(tools_dir)` (pure Rust AST), preserve cached dynamic entries *and* cached third-party entries (no re-glob) | dispatch: yes, tab: no |
| `ThirdPartyDrift`   | `build_static_manifest_with_venv(tools_dir, venv_dir)` (AST + glob + merge), preserve cached dynamic entries (third-party entries come from the fresh glob) | dispatch: yes, tab: no |

Both rebuild branches are entirely Rust. The Python introspection
subprocess is never spawned on dispatch.

After either rebuild branch, the new `static_hash` and
`third_party_hash` are stamped into the result before persisting.

### Wiring

A new function `ensure_manifest_fresh` lives in
`crates/toolr/src/bootstrap.rs` (next to
`ensure_manifest_present_or_bootstrap`):

```rust
pub(crate) fn ensure_manifest_fresh(
    cwd: &Path,
    argv: &[String],
) -> anyhow::Result<()> {
    let Ok(root) = discover_project_root(cwd) else {
        return Ok(());
    };
    let tools = root.join("tools");
    if !tools.join("pyproject.toml").is_file() {
        return Ok(());
    }
    if should_skip_auto_rebuild(argv) {
        return Ok(());
    }
    // Resolve venv if available; freshness is venv-tolerant.
    let venv_dir = resolve_venv_path(&root).ok().map(|r| r.venv_dir);

    let manifest_path = tools.join(".toolr-manifest.json");
    let cached = load_manifest(&manifest_path).ok();
    match freshness::compare(cached.as_ref(), &tools, venv_dir.as_deref())? {
        FreshnessVerdict::Fresh => Ok(()),
        verdict => match try_rebuild(verdict, &tools, venv_dir.as_deref(), cached) {
            Ok(fresh) => persist(&manifest_path, &fresh),
            Err(e)   => warn_and_keep_cache(e),
        },
    }
}
```

`main.rs::run` calls it directly after the existing
`ensure_manifest_present_or_bootstrap`:

```rust
bootstrap::ensure_manifest_present_or_bootstrap(&cwd, &argv)?;
bootstrap::ensure_manifest_fresh(&cwd, &argv)?;   // NEW
let manifest = load_or_empty(&cwd);
```

`load_or_empty` itself stays a plain reader — no freshness logic
leaks into call sites that don't have argv context.

The same `should_skip_auto_rebuild` filter that gates
`ensure_manifest_present_or_bootstrap` gates the freshness step.
That guarantees `toolr __complete …`, `toolr self …`,
`toolr project …`, `toolr init`, and `toolr --version` all bypass
the check, preserving their tight latency budgets.

### Tab-completion path

`complete::resolve_manifest_at_tab` is refactored to call
`freshness::compare` and act on the same verdict, but with two
behavioral differences from dispatch:

- **No persistence.** Tab completion stays purely in-memory.
  Concurrent completion processes won't race each other on disk
  writes.
- **No `ThirdPartyDrift` rebuild.** Tab completion already only
  re-runs the static AST pass when `static_hash` differs (see
  current `freshness.rs:32-67`). Keep that behavior — the
  completion hot path is too tight to glob + read venv files on
  every Tab press.

The shared `compare` function still returns the verdict; tab
completion just treats `ThirdPartyDrift` the same as `Fresh` for
its in-process result (the cached third-party entries are still
correct enough for completion).

### Error handling

| Failure                                                | Behavior                                                                                                    |
|--------------------------------------------------------|-------------------------------------------------------------------------------------------------------------|
| `hash_tools_dir(tools/)` fails (permission, mid-walk)  | Hard error — dispatch cannot proceed without a healthy `tools/`.                                              |
| Venv glob fails (missing/broken venv)                   | Treat as `third_party_hash = ""`. `build_static_manifest_with_venv` is already venv-tolerant. No hard error. |
| Cached manifest file is corrupt JSON or wrong schema    | Treat as `cached = None`. Verdict becomes `ThirdPartyDrift` (forces full rebuild). Same path as the existing `ensure_manifest_present_or_bootstrap` "missing" branch. |
| Static rebuild itself fails (`BuildError`)              | **Soft-fail.** Print two-line warning naming the offending file/symbol, keep using cached manifest. Direct the user to `toolr project manifest rebuild` for the full structured diagnostic. |

The warning leverages the attribution already in `BuildError`:

```text
toolr: warning: tools manifest is stale and a fresh build failed; falling back to cached manifest
toolr: warning: cause: parsing /Users/.../tools/jenkins/job.py: invalid syntax at line 42
toolr: warning: run `toolr project manifest rebuild` to see the full error
```

`BuildError::Build(anyhow::Error)` already wraps file-level failures
with `reading {path}` / `parsing {path}` (`parser/mod.rs:33-35`).
`BuildError::UnsupportedTypes(Vec<TypeResolutionError>)` and
`BuildError::UnknownGroupRefs(Vec<UnknownGroupRef>)` carry per-symbol
attribution including `module`, `command`, and a Levenshtein suggestion
(`parser/build.rs:204-216`, `220-233`). `BuildError::ThirdParty(ThirdPartyError)`
covers per-file failures inside `site-packages/<pkg>/toolr-manifest.json`.
`BuildError::Argparse(ArgparseError)` covers the argparse scanner.
For a soft-fail warning, the cause line uses `BuildError::Display` and
truncates to the first failure when the list form is more than one entry.

## Removals (entry-point deprecation)

The legacy `toolr.commands` entry-point mechanism is removed
entirely. Third-party plugins must ship `toolr-manifest.json`
in their installed package root.

**Python (`crates/toolr-py/python/toolr/_introspect.py`):**

- Delete `_load_entry_points` (current lines 127-148).
- Remove its call from `build_payload`.
- Remove the rationale comment about "Task 4's entry-point pass
  for legacy third-party packages".

**Python testing (`crates/toolr-py/python/toolr/testing.py`):**

- Remove the `skip_loading_entry_points` field, `entry_points_patcher`,
  `_default_entry_points_patcher`, and the `_load_entry_points` call
  from the test harness (current lines 42-90 region).

**Rust (`crates/toolr-core/src/dynamic/`):**

- `hash.rs` — replace `compute_dynamic_hash(venv_root)` with
  `compute_third_party_hash(venv_root)` that hashes the
  `glob_manifests(venv)` result (sorted paths + blake3 of each
  file's content).
- `merge.rs`, `rebuild.rs` — update field references
  `dynamic_hash` → `third_party_hash`.
- `Manifest` model: rename field `dynamic_hash: String` →
  `third_party_hash: String`. No `SCHEMA_VERSION` bump.

## Documentation

### UNRELEASED.md

Add a "Breaking" subsection noting:

- The `toolr.commands` entry-point mechanism is removed.
  Third-party plugins must ship a static `toolr-manifest.json`
  in their installed package root.
- Migration: run `toolr self build-manifest <package_name>` in
  the plugin repo, include the resulting `toolr-manifest.json`
  in the built wheel (e.g., via `MANIFEST.in` or your build
  backend's wheel-include configuration), and delete the now-
  inert `[project.entry-points.'toolr.commands']` entry from
  `pyproject.toml`.

### docs/third-party.md (or a new docs page if the existing one is too small)

New or expanded sections covering the plugin-author lifecycle:

1. **Why a JSON manifest** — toolr's CLI is a Rust binary that
   boots in <50 ms. Command discovery globs `toolr-manifest.json`
   at the root of every installed Python package in the tools
   venv. Plugin commands must be statically declared in this file.
2. **Generating the manifest** — `toolr self build-manifest
   <package_name>` writes the manifest to the package root.
3. **Shipping the manifest** — include the file in your wheel.
   Concrete example for at least one common build backend
   (hatchling: `[tool.hatch.build.targets.wheel] include = [...]`).
4. **Keeping it in sync** — `toolr self build-manifest <pkg> --check`
   exits non-zero on drift. Wire as a pre-commit hook and as a
   CI check.
5. **What happens if you skip it** — commands won't appear in
   `toolr --help` or `toolr <group> --help`. Verify by checking
   the installed package: `python -c "import <pkg>; print(<pkg>.__path__)"`
   should contain `toolr-manifest.json`.
6. **Migration from entry-point plugins** — mirrors the
   UNRELEASED.md note with a worked example.

## Testing strategy

### Unit — `freshness::compare`

- `Fresh` when both hashes match the cached manifest.
- `StaticDrift` when a `.py` file is added / removed / modified
  in `tools/`.
- `ThirdPartyDrift` when a `toolr-manifest.json` is added /
  removed / modified in any `site-packages/*/`.
- **Negative test (the user-visible win):** modifying an
  unrelated `.dist-info` (no `toolr-manifest.json` touched)
  returns `Fresh`.

### Unit — rebuild routing

- `StaticDrift` triggers `build_static_manifest`, not the
  `_with_venv` variant; the resulting manifest still contains
  the cached third-party entries (preserve invariant).
- `ThirdPartyDrift` triggers `build_static_manifest_with_venv`;
  cached third-party entries are replaced from the live glob.
- Both branches stamp the new `static_hash` and
  `third_party_hash` before persistence.

### Unit — `compute_third_party_hash`

- Adding a `toolr-manifest.json` changes the hash.
- Removing one changes the hash.
- Modifying content (same path) changes the hash.
- Touching a `.dist-info` directory without changing any
  `toolr-manifest.json` does *not* change the hash.

### Integration — dispatch happy path

Build a fake project with a `tools/example.py`, prime the manifest
*without* `example.py`, then add the file and dispatch `toolr
example --help`. Assert that the command resolves and that
`tools/.toolr-manifest.json` on disk now contains the `example`
group. This is the dashtastic-style regression test.

### Integration — skip list

For each argv that `should_skip_auto_rebuild` returns true on
(`__complete`, `self …`, `project …`, `init`, `--version`),
confirm dispatch does not invoke `freshness::compare` (e.g., by
making `tools/` un-hashable and asserting these still succeed).

### Integration — soft-fail

Introduce a syntax error in an unrelated `tools/*.py`, dispatch
`toolr <user-cmd>`. Assert: stderr contains the warning naming
the offending file, the cached manifest is still used, and the
user command's resolution proceeds (or fails on its own merits).

### Tab-completion regression

`resolve_manifest_at_tab` continues to operate in-memory only;
no `.toolr-manifest.json` write occurs as a side effect of any
completion invocation. Also confirm `ThirdPartyDrift` returned
by `compare` does not trigger a `_with_venv` rebuild on the tab
path (cost guard).
