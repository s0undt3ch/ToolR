    Project configuration¶

    Every repo using toolr has a tools/pyproject.toml. It declares the Python dependencies your commands need (the toolr package itself, plus anything your tools/*.py files import) and a handful of toolr-specific options.

    A minimal file:

    [project]
name = "tools"
version = "0.0.0"
requires-python = ">=3.11"
dependencies = [
    "toolr",
]

[tool.toolr]
venv-location = "cache"

    That's enough for toolr project venv sync to materialise a working venv and for the binary to dispatch commands.

    [project] block¶

    Standard PEP 621 metadata. Toolr inspects:

    • requires-python — used as a fallback for the venv's Python version when [tool.toolr] python-version is not set.
    • dependencies — passed straight through to uv sync. Add every package your tools/*.py files import. The toolr package itself must be present — toolr refuses to operate against a venv that doesn't have it installed.

    Other PEP 621 fields (name, version, description, …) are informational; toolr doesn't read them.

    [tool.toolr] options¶

    venv-location¶

    Where the tools venv is materialised.

    • Default: "cache" — $XDG_CACHE_HOME/toolr/<repo-key>/venv/ (per-repo, shared across worktrees of the same checkout, easy to prune via toolr self cache prune).
    • Alternative: "in-tree" — tools/.venv/ inside the repo. Useful when the cache directory is volatile (CI runners, ephemeral containers) or you want the venv visible alongside the source.
    [tool.toolr]
venv-location = "in-tree"

    The default tools/.gitignore written by toolr project init already excludes .venv/, so the in-tree venv won't leak into git.

    Setting the environment variable TOOLR_VENV_LOCATION overrides the value in tools/pyproject.toml. Accepts the same in-tree / cache spellings the TOML key does. Useful as a CI escape hatch — the s0undt3ch/ToolR setup-toolr action sets it to in-tree automatically so workflows can cache tools/.venv directly without the per-repo config caring.

    editable-install¶

    A list of paths to uv pip install -e after the initial sync. Typically used to pick up an editable install of the surrounding project so commands can import the_app directly.

    [tool.toolr]
editable-install = ["."]

    Each entry is resolved relative to the repo root. The post-sync hook is best-effort — failures log a warning but don't fail the sync.

    python-version¶

    Explicit override of the venv's Python version. Falls back to [project] requires-python when unset.

    [tool.toolr]
python-version = "3.13"

    Use this when your tools need a newer Python than your application code, or vice versa.

    tools/uv.lock¶

    Created by the first uv sync invocation. Commit it to git — that's how everyone on the team ends up with the same resolved versions.

    Regenerated by any of:

    • toolr project venv sync — full sync (re-resolve + apply).
    • toolr project venv sync -U — re-resolve every package to its latest allowed version, then apply.
    • toolr project venv sync -P <pkg> — re-resolve a specific package (repeatable), then apply.
    • toolr project venv lock — re-resolve without touching the venv (wraps uv lock). Also accepts -U / -P <pkg>.
    • uv lock directly inside the tools/ directory — same as above but bypasses toolr's freshness stamp.

    Interaction with toolr project venv sync¶

    The sync flow is:

    1. Discover or install uv.
    2. Resolve the venv location (venv-location + repo-key, or tools/.venv/).
    3. Materialise / refresh the venv with uv sync.
    4. Apply each entry in editable-install via uv pip install -e.
    5. Verify the toolr Python package is present in the venv; refuse to proceed otherwise.

    toolr project init runs this same flow automatically after writing the scaffold (skippable with --no-sync).

    Managing dependencies¶

    The recommended way to add or remove Python packages from tools/pyproject.toml is through the venv add and venv remove subcommands — they update the file and sync the venv in one step:

    # Add a package (spec formats: name, name@version, name>=1.2, …)
toolr project venv add requests
toolr project venv add "httpx>=0.27"

# Remove a package (must already appear in tools/pyproject.toml)
toolr project venv remove requests

    Both commands call through to uv add / uv remove and then run the sync flow, so the lock file and the venv stay consistent.

    To upgrade packages without adding new ones, use venv sync -U (all) or venv sync -P <pkg> (specific):

    # Upgrade everything to the latest allowed versions
toolr project venv sync -U

# Upgrade just httpx
toolr project venv sync -P httpx

    To re-lock without immediately applying (e.g. for review before committing), use venv lock:

    toolr project venv lock -U        # re-resolve all; don't touch the venv
toolr project venv lock -P httpx  # re-resolve httpx only

    Validating your configuration¶

    A quick sanity check:

    toolr project venv path

    Prints the absolute path where toolr expects the venv to live. If the printed path doesn't match the directory you intend (cache vs in-tree), revisit venv-location.
